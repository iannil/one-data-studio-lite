# 文档规范

## 目录结构

```
docs/
├── progress/           # 进行中的修改进展
├── reports/            # 验收和完成报告
│   └── completed/      # 已完成的报告归档
├── standards/          # 文档规范和标准
├── templates/          # 文档模板
├── PROJECT_STATUS.md   # 项目状态总览 (LLM 友好)
└── ISSUES.md           # 待解决问题清单
```

## 命名规范

### 进展文档 (docs/progress/)

- 格式: `{功能描述}-{YYYY-MM-DD}.md`
- 示例: `验收进展-2026-02-16.md`, `ETL优化-2026-02-17.md`

### 完成报告 (docs/reports/completed/)

- 格式: `{报告类型}-{YYYY-MM-DD}.md`
- 示例: `验收报告-2026-02-16.md`, `重构报告-2026-02-17.md`

### 验收报告 (docs/reports/)

- 格式: `{子系统}验收-{YYYY-MM-DD}.md`
- 示例: `ETL验收-2026-02-16.md`

## 文档状态标记

| 标记 | 含义 |
|------|------|
| 📝 进行中 | 文档正在编写或任务正在执行 |
| ✅ 已完成 | 任务已完成，文档已归档 |
| ⚠️ 待处理 | 发现问题，等待解决 |
| ❌ 已废弃 | 文档不再有效，已过期 |

## 文档生命周期

1. **创建**: 在 `docs/progress/` 创建进展文档
2. **更新**: 执行过程中实时更新，附带时间戳
3. **完成**: 任务完成后移动到 `docs/reports/completed/`
4. **归档**: 更新状态为"已完成"并添加完成时间

## 内容规范

### 必须包含

- 明确的标题和日期
- 状态标识 (进行中/已完成)
- 执行步骤或结论
- 时间戳记录

### 禁止内容

- 重复的、冗余的信息
- 过期且未更新的数据
- 不反映实际情况的内容

## LLM 友好规范

为便于大模型理解和处理，文档应：

1. 使用结构化格式 (表格、列表)
2. 保持简洁清晰的描述
3. 提供关键文件路径索引
4. 使用统一的术语和命名
